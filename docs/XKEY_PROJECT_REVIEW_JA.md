# xKey プロジェクトレビューレポート

レビュー日: 2026-06-16
現在のバージョン: 5.7.0
スコープ: React/Vite ソースコード、Capacitor Android、ストレージ、セキュリティ、UI、多言語、ビルド、および製品の方向性。

## 1. プロジェクトの目的

xKey は、オフラインファーストの Web3 ウォレット保管 (vault) 管理アプリケーションです。このアプリを使用すると、ユーザーはウォレットアドレス、秘密鍵、シードフレーズ、メモ、ラベル、フォルダ、QR コード、`.xkey` バックアップファイル、CSV データ、資産残高をデバイスのローカルに保存できます。

xKey の中心的な目標は、オンラインの取引ウォレットではなく、ローカルの「秘密鍵の保管庫」になることです。ユーザーは xKey を使用して以下を行うことができます。

- 暗号化された保管庫で複数の Web3 ウォレットを管理する。
- 秘密鍵とシードフレーズをローカルの暗号化された形式で保存する。
- 新しいウォレットの作成、ウォレットの手動インポート、プレフィックス/サフィックスによるカスタム (vanity) ウォレットの生成。
- フォルダ、ラベル、ネットワーク、ピン留め状態、または残高によるウォレットのグループ化。
- パスワードで保護された `.xkey` ファイルを使用したバックアップ/復元。
- インベントリや監査が必要な場合の CSV のエクスポート。
- アドレスやウォレットデータの QR コードのスキャン、表示、共有、およびダウンロード。
- `$`, `USDT`, `VND`, `CNY`, `KRW`, `JPY`, `EUR`, `RUB`, `INR`、ポイント、またはカスタムラベルなどの任意の単位で残高を手動で追跡する。
- Android Device Credential を使用して、デバイスの指紋、顔、PIN、パスワード、またはパターンで保管庫のロックを解除する。

## 2. 現在の強み

### セキュリティとストレージ

- ウォレットデータはローカルで暗号化されており、オフライン保管庫の目的に合致しています。
- 秘密鍵やシードフレーズなどの機密フィールドは、ウォレットリスト全体が暗号化される前に、フィールドレベルでさらに暗号化されます。
- Android ネイティブには専用の Device Credential プラグインがあり、Android Keystore を使用して保管庫の鍵をラップしています。
- `android:allowBackup="false"` が AndroidManifest に設定されており、意図しないアプリデータのバックアップのリスクを軽減しています。
- 非アクティブ時の自動ロック、クリップボードの自動クリア、アプリが非アクティブなときのプライバシーシールド、機密データ表示時のマスターパスワードのプロンプトなどのメカニズムが含まれています。
- 保管庫に重大なエラーが発生した場合のワイプ/リセット機能をサポートしています。

### ユーザーエクスペリエンス

- ホームページはレスポンシブレイアウトをサポートしており、大画面でのマルチカラムのウォレットリストやモバイル向けの最適化を備えています。
- 5% から 200% までのカスタマイズ可能なディスプレイスケーリングを備えており、小さなデバイスや、より多くのデータを表示したいユーザーに適しています。
- ウォレットリストの dense/compact/ultra compact モードを提供します。
- コピー、QR、ウォレットの展開、ウォレットの追加、ツール、検索、フィルター、並べ替えのボタンは、実際のワークフローに近い場所に配置されています。
- カスタム (vanity) ウォレットフォルダ、NEW ラベル、新しく作成されたウォレットの光るリング、および新しいウォレットを含むフォルダへの自動ナビゲーションを備えています。
- 残高編集モーダルには、検索、貼り付け、アドレスのコピー、CSV インポート、フィルター、下書きの自動保存が含まれています。
- トースト/確認画面はよりプロフェッショナルに見えるように刷新され、表示比率に応じて拡大縮小する傾向があります。

### 機能

- 通常のウォレットの作成、手動インポート、および専用のワーカーを使用したカスタム (vanity) ウォレットの生成。
- `.xkey` バックアップ、CSV インポート/エクスポート、重複検出器、分析、高度なツール。
- パスワードで保護された QR 転送、QR スキャナー、QR 共有/ダウンロード。
- 人気のあるネットワークのサポート: XLAYER, ETH, BSC, Polygon, Arbitrum, Optimism, Solana, Tron, Base。
- 15 言語による多言語サポート。
- バージョンは `package.json`/ネイティブアプリ情報から取得され、アプリ内に表示されます。

### ビルドと Android

- `npm run lint` が正常に完了します。
- `npm run build` が正常に完了します。
- `npx cap sync android` が Web アセットを Android に正常に同期します。
- 現在の Android バージョンは `versionName "5.7.0"` および `versionCode 57` です。
- `.gitignore` は `1/`、ビルドアーティファクト、署名シークレット、`.xkey`、APK/AAB、およびローカルファイルを適切に除外しています。

## 3. 弱点と潜在的な問題

### 高レベル

1. 依存関係に `npm audit` からのセキュリティ警告があります。

   `npm audit --omit=dev` を実行すると、次の報告があります。

   - `vite 8.0.0 - 8.0.15`: 高い重大度、開発サーバーの Windows パス/UNC に関連。
   - `ethers` 経由の `ws`: 高/中程度の重大度。`npm audit fix --force` は `ethers` をメジャーバージョン 5 にダウングレードすることを提案していますが、これは破壊的変更 (breaking changes) を引き起こす可能性があります。

   推奨事項: まず、パッチ/マイナーの範囲内で安全に Vite を更新してください。`ethers/ws` については、新しい `ethers` バージョンを確認するか、アップストリームでサポートされている場合は `ws` をオーバーライドします。盲目的に `--force` を使用することは避けてください。

2. Android リリース版で shrink/minify が有効になっていません。

   現在、`android/app/build.gradle` には `release { minifyEnabled false }` が含まれています。これによりアプリがクラッシュすることはありませんが、APK/AAB のリバースエンジニアリングが容易になり、サイズが大きくなります。

   推奨事項: リリース版で R8/ProGuard を有効にし、必要に応じて Capacitor/プラグインの keep rules を追加し、公開前に徹底的にテストすることを検討してください。

3. フォールバック AES キーが依然として Preferences に保存されています。

   コードは現在、リカバリまたは Web/フォールバックの互換性のために `xkey_aes_fallback` を保存しています。これは、デバイスのロック方法を変更したときに保管庫が失われるリスクを軽減するためのトレードオフですが、ネイティブ Android のセキュリティの観点からは、キーを Keystore のみに保持するよりも脆弱です。

   推奨事項: 2 つのモードを明確に分離します。
   - Android セキュアモード: キーは Keystore/Device Credential を介してのみアンラップされます。
   - 互換性モード: フォールバックキーを保持し、ユーザーに明確な警告を表示します。

4. 一部の副次的な翻訳にまだ英語の文字列が含まれています。

   自動チェックによると、`de`、`fr`、`es`、`hi`、`id`、`pt`、`tr`、`ar`、`th` などの多くのロケールに、`Remove master password?`、`Enter master password`、`Wrong password`、`Pinned`、`Unpin`、`Double AES-256 with biometrics` などの文字列が残っています。

   推奨事項: ロケールにキーが欠落している場合、または重要な生のキー (raw keys) がまだある場合にビルドを失敗させる i18n チェックスクリプトを CI で作成します。

### 中レベル

1. 複数の言語のロケールでキーが不足しています。

   `en.js` と比較して、`vi` 以外のほとんどのロケールで以下の項目が不足しています。
   - `common.warning`
   - `createWallet.vanityLongTitle`

   `LanguageContext` は英語にフォールバックするため、アプリはクラッシュしませんが、多言語エクスペリエンスが不完全です。

2. 多くのロケールで `chainBulk` が不要なキーとして存在しています。

   多くのロケールには `chainBulk.*` グループがありますが、`en.js` にはありません。これらはレガシーキーまたは同期されていないキーの可能性があります。直接エラーを引き起こすことはありませんが、翻訳の管理が困難になります。

3. CryptoJS AES パスフレーズモードは最新の暗号化標準ではありません。

   `CryptoJS.AES.encrypt(data, key)` は機能しますが、専用の salt/KDF/IV/auth タグを持つ標準モデルほど明確ではありません。AES-GCM または WebCrypto の方が監査が容易になります。

   長期的な推奨事項: 明確に定義された PBKDF2/Argon2id パラメータ、バージョン管理されたペイロード、および必須の認証タグを使用して、保管庫のフォーマットを WebCrypto AES-GCM に移行します。

4. マスターパスワードは 10,000 回の反復処理の PBKDF2 を使用しています。

   このレベルは、機密データを保護するための今日の基準からするとやや低いです。これは秘密鍵/シードを表示するための 2 次パスワードであり、メインの保管庫キーではありませんが、増やす必要があります。

   推奨事項: デバイスのベンチマークに基づいて反復回数を増やし、移行によって古いデータが破損しないようにバージョンのハッシュを保存します。

5. クリップボードの自動クリアは、すべてのプラットフォームで完全に保証されているわけではありません。

   コードはクリアする前にクリップボードに正しい値が保持されているかチェックしますが、これは良いアプローチです。ただし、ユーザーのジェスチャによってトリガーされない場合、Android/ブラウザはクリップボードへの書き込みを制限する可能性があります。

   推奨事項: 絶対的な約束をせず、「OS が許可する場合、xKey はクリップボードのクリアを試みます」と UI に明確に記述してください。

6. 実用的な自動テストの欠如。

   プロジェクトには lint/ビルドがありますが、ロック解除、インポート/エクスポート、カスタムウォレットの作成、残高の編集、クリップボードの操作、i18n などの重要なフローのユニット/e2e テストがありません。

   推奨事項: Web 用に Playwright を使用したスモークテストを追加し、Android のインストルメンテーション/手動リリーステストのチェックリストを追加します。

### 低レベル

1. `console.error` がいくつかの場所に残っています。

   致命的ではありませんが、本番環境で不要なスタックトレースが漏洩するのを防ぐために、ロガーまたは開発専用環境にグループ化する必要があります。

2. Vite が巨大なチャンクに関する警告を出します。

   `index` と `scan` チャンクが大きいです。これはランタイムエラーではありませんが、ローエンドデバイスでのアプリの読み込みが遅くなる可能性があります。

   推奨事項: QR スキャナー、ethers を多用するパス、高度なツール、ダッシュボードのより深いビューを遅延読み込み (Lazy load) します。

3. 非常に低い、または非常に高いスケール比率で一部の UI が崩れる可能性があります。

   アプリは多くの領域でスケーリングをうまく処理していますが、大きなモーダル、QR コード、密なフォーム、ボトムシート、ウォレットカードについては、引き続き 5%、50%、75%、100%、150%、200% でテストする必要があります。

## 4. 機能グループの評価

### ロック解除のセキュリティ

OS に生体認証を処理させ、PIN/パスワード/パターンにフォールバックできるため、Android Device Credential アプローチは正しいです。主なリスクは、古い PIN メカニズム、フォールバックキー、および Keystore キー間の移行にあります。

推奨事項:
- 「保管庫のセキュリティステータス」画面を提供する: Android Secure、Web Fallback、Compatibility (デバイスロックのセットアップが必要)。
- 無効になったキーが検出された場合、古い保管庫に暗号文がまだ含まれている場合は、新しい保管庫キーを自動生成しないでください。代わりに、ユーザーをリカバリ/ワイプに誘導します。
- 機密データを記録せずに、内部のロック解除ステータスを記録します。

### カスタム (Vanity) ウォレット生成

UI のフリーズを防ぐため、別のワーカーを使用するのは正しいアプローチです。ウォレットの数量、フォルダへの自動保存、生成中の自動ロックの停止、時間制限、長すぎるパターンの警告など、最近のアップグレードはすべて理にかなっています。

推奨事項:
- パターンの長さに基づいて、確率/推定時間を明確に表示します。
- ジョブの一時停止/再開/停止を許可します。
- ユーザーがどのウォレットがどのバッチのものかを把握できるように、生成されたジョブの履歴を保存します。
- モバイルで過度に長いパターンには強力な警告を提供します。

### 資産残高の編集

現在のワークフローは、ブロックチェーンエクスプローラーでアドレスを確認し、残高を手動で入力するユーザーに適しています。長所には、検索、アドレスのコピー、貼り付け、フィルター、CSV サポート、下書きの自動保存などがあります。

推奨事項:
- 「ステップバイステップの検証」モードを追加する: 画面に一度に 1 つのウォレットを表示し、完全なアドレス、コピーボタン、ネットワーク別のエクスプローラーへのリンク、大きな入力フィールドを表示します。
- エントリのスキップを防ぐために「確認済み (checked)」としてマークできるようにします。
- `address,balance,unit,network` 列を持つ CSV インポートを許可します。
- 最後の編集に対する元に戻す (undo) 機能を追加します。

### 多言語

英語へのフォールバックにより UI が壊れるのを防ぎますが、海外のユーザーをターゲットとする製品には、より厳格な翻訳管理が必要です。

推奨事項:
- `npm run i18n:check` スクリプトを作成します。
- UI に欠落しているキー、不要なキー、生の翻訳キーを報告します。
- セキュリティ、バックアップ、ワイプ、秘密鍵、シードフレーズのグループに関する正確な翻訳を優先します。

### Android リリース

現在の構成はビルドと同期には十分ですが、リリース版の堅牢化 (hardening) が不足しています。

推奨事項:
- テスト後、リリース用に minify を有効にします。
- 明確な許可リストを使用して `npm audit --omit=dev` CI ステップを追加します。
- タグプッシュ時に GitHub Actions 経由で APK/AAB をビルドします。
- バージョン管理されたリリースノートをリポジトリに保管します。

## 5. 提案されるアップグレードのアイデア

### 短期

- すべての欠落している翻訳キーを修正します: `common.warning`、`createWallet.vanityLongTitle`。
- 他のロケールに残っている英語の文字列をクリーンアップします。
- CI に i18n チェックスクリプトを追加します。
- 現在のセキュリティ勧告を解決するために Vite を更新します。
- 設定に「セキュリティステータス」ページを追加します。
- クリップボードの自動クリアはベストエフォート (best-effort) であるという明確なメモを追加します。
- 残高編集モーダルに、ネットワーク別の「エクスプローラーで開く」ボタンを追加します。
- ウォレットの削除、残高の編集、フォルダの変更に対して、元に戻す (undo) スナックバーを追加します。

### 中期

- 暗号化フォーマットをバージョン管理された WebCrypto AES-GCM に移行します。
- Android セキュアモードと互換性モードを分離します。
- メインフローの Playwright スモークテストを追加します。
- スキャナー/分析/高度なツールを遅延読み込み (Lazy load) して、初期バンドルサイズを縮小します。
- 機密データを除外した設定のエクスポート/インポート機能を追加します。
- 「保管庫監査」モードの追加: バックアップがないウォレット、重複したアドレス、ネットワークがない、名前がない、または秘密鍵がアドレスと一致しない。

### 長期

- デバイスの変更、画面ロックの変更、生体認証の喪失、`.xkey` ファイルの紛失などのシナリオに対する公式のリカバリガイドを作成します。
- マルチパートの QR コードまたは一時ファイルを介した、暗号化されたマルチデバイス転送を追加します。
- セキュリティを重視するユーザー向けに、ハードウェアベースのみのオプションを追加します。
- チェックサム/ネットワークによるアドレス検証を追加します。
- 紙のバックアップテンプレートを提供する: アドレス、ネットワーク、メモ (ユーザーが選択した場合は秘密鍵を除く)。
- xKey をデスクトップのオフライン保管庫として使用したい場合の、より優れたデスクトップ/PWA サポート。

## 6. 将来の製品の方向性

xKey は「多くのウォレットを持つユーザー向けのプロフェッショナルなオフライン保管庫」の道を追求すべきです。オンラインのトランザクションウォレットに時期尚早に移行するべきではありません。これにより、セキュリティリスク、RPC への依存、フィッシングの媒介、トランザクション署名の責任、攻撃対象領域が増加するためです。

適切な方向性:
1. データセキュリティの優先: バックアップ、復元、移行、明確な警告、保管庫の監査。
2. 多くのウォレットの迅速な管理を優先: フォルダ、タグ、フィルター、一括編集、CSV、QR、カスタム (vanity) 生成。
3. 安定した Android ネイティブ機能の優先: Device Credential、Keystore、クリップボード管理、ファイルピッカー、QR の共有/ダウンロード。
4. 密度が高いが明確な UI を優先: スケーリング、コンパクトモード、レスポンシブなタブレットレイアウト、短いトースト、ブロックされないモーダル。
5. 透明性の優先: セキュリティステータス、リリースノート、アプリ内での明示的なバージョン管理、バックアップとワイプのガイド。

## 7. 結論

プロジェクトの基盤は強固です。機能が豊富で、明確なオフラインファーストのアプローチを持ち、Android Credential の統合は軌道に乗っており、UI はモバイル/タブレット向けに高度に最適化され、包括的なウォレット管理ツールセットを誇っています。

今後の最優先事項は、新しい機能を多数追加することではなく、アプリを「壊れにくくする (harder to break)」ことです。
- i18n の実装を完了します。
- Android リリース版を堅牢にします。
- Keystore/フォールバックのセキュリティモデルを明確にします。
- 重要なフローに自動テストを追加します。
- 依存関係の監査を管理します。
- 長期的に暗号化フォーマットを標準化します。

これらの点が適切に処理されれば、xKey は複数の Web3 ウォレットを管理するユーザーにとって非常に信頼性の高いオフライン保管庫ツールへと成長する可能性があります。
