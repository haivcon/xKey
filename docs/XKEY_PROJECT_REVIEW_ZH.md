# xKey 项目审查报告

审查日期: 2026-06-16
当前版本: 5.7.0
范围: React/Vite 源代码、Capacitor Android、存储、安全性、UI、多语言、构建及产品方向。

## 1. 项目用途

xKey 是一款以离线为主的 Web3 钱包保险库管理应用程序。该应用允许用户在本地设备上保存钱包地址、私钥、助记词、备注、标签、文件夹、二维码、`.xkey` 备份文件、CSV 数据以及资产余额。

xKey 的核心目标是成为一个本地使用的“私钥保险库”，而非在线交易钱包。用户可以使用 xKey 进行：

- 在加密保险库中管理多个 Web3 钱包。
- 以本地加密格式保存私钥和助记词。
- 创建新钱包、手动导入钱包、通过前缀/后缀生成靓号钱包。
- 按文件夹、标签、网络、置顶状态或余额对钱包进行分组。
- 使用带有密码的 `.xkey` 文件进行备份/恢复。
- 在需要清点或核对时导出 CSV。
- 扫描、显示、分享和下载钱包地址或数据的二维码。
- 手动跟踪资产余额，支持自选单位，如 `$`, `USDT`, `VND`, `CNY`, `KRW`, `JPY`, `EUR`, `RUB`, `INR`, 积分或自定义标签。
- 使用 Android Device Credential 通过设备的指纹、面部识别、PIN码、密码或图案来解锁保险库。

## 2. 当前优势

### 安全与存储

- 钱包数据在本地加密，符合离线保险库的目标。
- 诸如私钥和助记词等敏感字段在对整个钱包列表进行加密之前，会在字段级别进行额外加密。
- Android 原生版本拥有专用的 Device Credential 插件，使用 Android Keystore 封装保险库密钥。
- 在 AndroidManifest 中设置了 `android:allowBackup="false"`，降低了应用数据被意外备份的风险。
- 具备不操作时自动锁定、自动清除剪贴板、应用后台时开启隐私遮罩、查看敏感数据时要求输入主密码等机制。
- 当保险库出现严重错误时，支持擦除/重置功能。

### 用户体验

- 首页支持响应式布局，在宽屏上显示多列钱包列表，并针对移动端进行了优化。
- 支持 5% 到 200% 的自定义显示比例，适合小屏幕设备或希望查看更多数据的用户。
- 钱包列表提供密集/紧凑/超紧凑模式。
- 复制、二维码、展开钱包、添加钱包、工具、搜索、过滤和排序按钮的布局贴近实际工作流。
- 具有靓号钱包文件夹、NEW 标签、新创建钱包的发光环，并自动导航到包含新钱包的文件夹。
- 编辑余额的模态框包含了搜索、粘贴、复制地址、导入 CSV、过滤和草稿自动保存功能。
- Toast/Confirm 提示已重新设计，更加专业，并倾向于根据显示比例进行缩放。

### 功能特性

- 使用独立 worker 创建常规钱包、手动导入和生成靓号钱包。
- `.xkey` 备份、CSV 导入/导出、重复项检测器、分析、高级工具。
- 带密码保护的二维码传输、二维码扫描器、二维码分享/下载。
- 支持流行的网络：XLAYER, ETH, BSC, Polygon, Arbitrum, Optimism, Solana, Tron, Base。
- 支持 15 种语言的多语言功能。
- 从 `package.json`/原生应用信息中获取版本并在应用内显示。

### 构建与 Android

- `npm run lint` 运行成功。
- `npm run build` 运行成功。
- `npx cap sync android` 已成功将 web 资源同步到 Android。
- 目前的 Android 版本是 `versionName "5.7.0"` 和 `versionCode 57`。
- `.gitignore` 已排除了 `1/`、构建产物、签名机密文件、`.xkey`、APK/AAB 以及本地文件。

## 3. 缺点与潜在问题

### 高优先级

1. 依赖项存在来自 `npm audit` 的安全警告。

   运行 `npm audit --omit=dev` 报告：

   - `vite 8.0.0 - 8.0.15`: 高危，与开发服务器中的 Windows path/UNC 有关。
   - `ws`（通过 `ethers` 引入）: 高/中危。`npm audit fix --force` 建议将 `ethers` 降级到主版本 5，这可能引起破坏性更新（breaking change）。

   建议：首先在安全的 patch/minor 范围内更新 Vite。对于 `ethers/ws`，检查是否有更新的 `ethers` 版本，或者如果上游支持，覆盖 `ws`；避免盲目使用 `--force`。

2. Android 发布版未启用 shrink/minify。

   `android/app/build.gradle` 目前设置为 `release { minifyEnabled false }`。这不会导致应用崩溃，但会使 APK/AAB 更容易被逆向工程，且文件体积更大。

   建议：尝试在 release 版中启用 R8/ProGuard，如有需要则为 Capacitor/插件添加 keep rules，并在发布前进行充分测试。

3. Fallback AES 密钥仍保存在 Preferences 中。

   代码当前保存了 `xkey_aes_fallback` 用于恢复或 web/降级兼容性。这是一种权衡，有助于降低更换设备锁时丢失保险库的风险，但在 Android 原生安全性方面，它比只将密钥保留在 Keystore 中要弱。

   建议：明确区分两种模式：
   - Android 安全模式：仅通过 Keystore/device credential 解封密钥。
   - 兼容模式：保留 fallback 密钥，并向用户显示清晰的警告。

4. 某些次要翻译中仍残留英文。

   自动检查显示，许多 locale 如 `de`, `fr`, `es`, `hi`, `id`, `pt`, `tr`, `ar`, `th` 仍然包含如 `Remove master password?`, `Enter master password`, `Wrong password`, `Pinned`, `Unpin`, `Double AES-256 with biometrics` 的字符串。

   建议：在 CI 中创建一个 i18n 检查脚本，当 locale 缺少密钥或仍保留重要的原始密钥时，使构建失败。

### 中优先级

1. 多种语言的 Locale 缺少密钥。

   与 `en.js` 相比，除 `vi` 之外的大多数 locale 缺失：
   - `common.warning`
   - `createWallet.vanityLongTitle`

   由于 `LanguageContext` 会回退到英语，应用不会崩溃，但多语言体验不够完整。

2. `chainBulk` 在许多 locale 中是多余的密钥。

   许多 locale 包含 `chainBulk.*` 组，但 `en.js` 没有。这些可能是旧密钥或未同步的密钥。它不会直接导致错误，但增加了翻译管理的难度。

3. CryptoJS AES passphrase 模式不是最现代的加密模型。

   `CryptoJS.AES.encrypt(data, key)` 能够工作，但不如具有专用 salt/KDF/IV/auth tag 的标准模型那么清晰。AES-GCM 或 WebCrypto 会更容易被审计。

   长期建议：将保险库格式迁移到 WebCrypto AES-GCM，具有明确的 PBKDF2/Argon2id 参数、版本化负载和强制的身份验证标签（authentication tag）。

4. 主密码使用了 10,000 次迭代的 PBKDF2。

   如果用于保护敏感数据，与目前的期望相比，这个级别略低。虽然它是用于查看私钥/助记词的辅助密码，而不是主保险库密钥，但仍应提高。

   建议：根据设备基准测试增加迭代次数，并保存版本哈希以避免在迁移时破坏旧数据。

5. 剪贴板自动清除无法在所有平台上得到绝对保证。

   代码在清除前已经检查剪贴板是否仍然保留正确的值，这是一个好的方向。但是，当不是由用户手势触发时，Android/浏览器可能会限制剪贴板的写入。

   建议：在 UI 中明确说明“xKey 将在操作系统允许的情况下尝试清除剪贴板”，不作绝对承诺。

6. 缺乏实际的自动化测试。

   项目有 lint/build，但尚未看到针对重要流程（如解锁、导入/导出、创建靓号钱包、编辑余额、剪贴板、i18n）的 unit/e2e 测试。

   建议：添加用于 Web 的 Playwright 冒烟测试，并制定 Android instrumentation/手动发布的检查清单。

### 低优先级

1. 少数地方仍有 `console.error`。

   不严重，但应汇总到 logger/dev-only 中，避免在生产环境中暴露不必要的堆栈信息。

2. Vite 提示大型 chunk 警告。

   `index` 和 `scan` chunk 较大。这不是运行时错误，但在低端设备上可能会导致应用加载变慢。

   建议：对二维码扫描器、重量级 ethers 路径、高级工具和更深层次的仪表板进行延迟加载（lazy load）。

3. 某些 UI 在缩放比例极低或极高时有错位的风险。

   应用已经缩放了许多部分，但是大模态框、二维码、密集表单、底部表单和钱包卡片仍需要在 5%、50%、75%、100%、150%、200% 时进行测试。

## 4. 按功能模块评估

### 解锁安全

Android Device Credential 的方向是正确的，因为它让操作系统处理生物识别和备用 PIN/密码/图案。主要风险在于旧 PIN 机制、fallback 密钥和 Keystore 密钥之间的迁移。

建议：
- 增加“保险库安全状态”屏幕：Android Secure、Web Fallback、Compatibility，需要设置设备锁。
- 检测到密钥失效时，如果旧保险库仍包含密文，则不要自动生成新的保险库密钥；只应引导用户进行恢复/擦除。
- 记录内部解锁状态日志，不记录敏感数据。

### 生成靓号钱包

使用独立的 worker 是正确的，因为它避免了 UI 卡顿。最近的升级，如钱包数量、自动保存到文件夹、生成期间暂停自动锁定、时间限制和过长模式警告，都是合理的。

建议：
- 根据模式长度清楚地显示概率/估计时间。
- 允许停止/暂停/继续任务。
- 保存已生成任务的历史记录，以便用户知道哪些钱包来自哪个批次。
- 在移动端上对过长模式提出强烈警告。

### 资产余额编辑

当前的工作流程适合那些在区块链浏览器上验证地址然后手动输入余额的用户。优势在于有搜索、复制地址、粘贴、过滤、CSV 和自动保存草稿功能。

建议：
- 增加“逐个核对”模式：屏幕只显示 1 个钱包、完整地址、复制按钮、按网络分类的浏览器链接和较大的输入框。
- 允许标记“已检查”以避免遗漏。
- 允许按 `address,balance,unit,network` 列导入 CSV。
- 为最近一次修改添加撤销功能。

### 多语言

回退到英语有助于避免应用界面崩溃，但面向多国的产品需要更严格的翻译控制。

建议：
- 创建 `npm run i18n:check` 脚本。
- 在 UI 中报告缺失的密钥、多余的密钥、原始翻译密钥。
- 优先准确翻译安全、备份、擦除、私钥、助记词相关的组。

### Android 发布

目前的配置足以进行构建和同步，但发布版本的加固还不够。

建议：
- 在测试后开启发布版的 minify。
- 添加具有明确许可列表的 CI 步骤 `npm audit --omit=dev`。
- 根据 tag 使用 GitHub Actions 构建 APK/AAB。
- 在代码库中按版本保存发布说明。

## 5. 建议的升级想法

### 短期

- 修复所有缺失的翻译密钥：`common.warning`, `createWallet.vanityLongTitle`。
- 清理其他 locale 中残留的英文字符串。
- 在 CI 中添加 i18n 检查脚本。
- 更新 Vite 以处理当前的 advisory。
- 在设置中添加“安全状态”页面。
- 为剪贴板自动清除添加明确注释（尽力而为）。
- 在编辑余额模态框中添加按网络的“在浏览器中打开”按钮。
- 为删除钱包、编辑余额、更改文件夹添加撤销 snackbar。

### 中期

- 将加密格式迁移到版本化的 WebCrypto AES-GCM。
- 区分 Android 安全模式和兼容模式。
- 为主要流程添加 Playwright 冒烟测试。
- 延迟加载扫描器/分析/高级工具，以减小初始 bundle。
- 添加不包含敏感数据的设置导出/导入功能。
- 添加“审计保险库”模式：缺少备份的钱包、重复地址、缺少网络、缺少名称、私钥与地址不匹配。

### 长期

- 建立官方恢复指南，适用于更换设备、更改屏幕锁、丢失生物识别、丢失 `.xkey` 文件等情况。
- 通过多段二维码或临时文件添加加密的多设备传输功能。
- 为需要高安全性的用户添加仅限硬件支持的选项。
- 添加通过校验和/网络进行的地址验证。
- 提供纸质备份模板：地址、网络、备注，若用户选择则不包含私钥。
- 如果希望在计算机上将 xKey 用作离线保险库，提供更稳定的桌面/PWA 支持。

## 6. 未来产品发展方向

xKey 应该沿着“为拥有众多钱包的用户提供专业离线保险库”的方向发展。不应过早将其转变为在线交易钱包，因为这会增加安全风险、RPC 依赖、钓鱼攻击向量、交易签名责任和攻击面。

合适的方向：
1. 数据安全优先：备份、恢复、迁移、明确的警告、保险库审计。
2. 管理众多钱包的快捷操作优先：文件夹、标签、过滤、批量编辑、CSV、二维码、靓号生成。
3. 稳定的 Android 原生功能优先：Device Credential、Keystore、剪贴板管理、文件选择器、分享/下载二维码。
4. 紧凑但清晰的 UI 优先：缩放、紧凑模式、响应式的平板布局、简短的 toast、非遮挡式模态框。
5. 透明度优先：安全状态、发布说明、应用内明确的版本信息、备份和擦除指南。

## 7. 结论

该项目有一个很好的基础：功能丰富，以离线为主的思路清晰，Android Credential 集成方向正确，UI 针对手机/平板进行了深度优化，并拥有丰富的钱包管理工具集。

接下来的首要任务不是添加大量新功能，而是让应用“更不容易出错”：
- 完善 i18n。
- 加固 Android 发布版本。
- 明确 Keystore/fallback 安全模型。
- 为关键流程添加自动化测试。
- 控制依赖项审查。
- 长期标准化加密格式。

如果处理好这些问题，xKey 可以发展成为一个高度可靠的离线保险库工具，供用户管理多个 Web3 钱包。
