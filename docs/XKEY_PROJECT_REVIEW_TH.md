# รายงานทบทวนโครงการ xKey

วันที่ทบทวน: 2026-06-16
เวอร์ชันปัจจุบัน: 5.7.0
ขอบเขต: ซอร์สโค้ด React/Vite, Capacitor Android, การจัดเก็บข้อมูล, ความปลอดภัย, UI, รองรับหลายภาษา, บิลด์ และทิศทางผลิตภัณฑ์

## 1. วัตถุประสงค์ของโครงการ

xKey เป็นแอปพลิเคชันการจัดการตู้นิรภัย (vault) สำหรับกระเป๋าเงิน Web3 ที่เน้นการใช้งานออฟไลน์เป็นหลัก (offline-first) แอปพลิเคชันช่วยให้ผู้ใช้สามารถจัดเก็บที่อยู่กระเป๋าเงิน, ไพรเวตคีย์ (private keys), วลีเมล็ดพันธุ์ (seed phrases), บันทึกย่อ, ป้ายกำกับ, โฟลเดอร์, คิวอาร์โค้ด (QR codes), ไฟล์สำรอง `.xkey`, ข้อมูล CSV และยอดคงเหลือสินทรัพย์ไว้ในอุปกรณ์โดยตรง

เป้าหมายหลักของ xKey คือการเป็น "ตู้นิรภัยสำหรับไพรเวตคีย์" แบบในตัวเครื่อง ไม่ใช่กระเป๋าเงินสำหรับทำธุรกรรมออนไลน์ ผู้ใช้สามารถใช้ xKey เพื่อ:

- จัดการกระเป๋าเงิน Web3 หลายใบในตู้นิรภัยที่มีการเข้ารหัส
- จัดเก็บไพรเวตคีย์และวลีเมล็ดพันธุ์ในรูปแบบที่เข้ารหัสในเครื่อง
- สร้างกระเป๋าเงินใหม่, นำเข้ากระเป๋าเงินด้วยตนเอง, สร้างกระเป๋าเงินที่ปรับแต่ง (vanity) โดยใช้คำนำหน้า/คำต่อท้าย
- จัดกลุ่มกระเป๋าเงินตามโฟลเดอร์, ป้ายกำกับ, เครือข่าย, สถานะการปักหมุด หรือยอดคงเหลือ
- สำรองข้อมูล/กู้คืน โดยใช้ไฟล์ `.xkey` ที่ป้องกันด้วยรหัสผ่าน
- ส่งออก CSV เมื่อจำเป็นต้องตรวจสอบสินค้าคงคลังหรือการตรวจสอบ (auditing)
- สแกน, แสดง, แชร์ และดาวน์โหลดคิวอาร์โค้ดสำหรับที่อยู่หรือข้อมูลกระเป๋าเงิน
- ติดตามยอดคงเหลือด้วยตนเองในหน่วยที่เลือกได้ เช่น `$`, `USDT`, `VND`, `CNY`, `KRW`, `JPY`, `EUR`, `RUB`, `INR`, คะแนน, หรือป้ายกำกับที่กำหนดเอง
- ใช้ Android Device Credential เพื่อปลดล็อกตู้นิรภัยด้วยลายนิ้วมือ, ใบหน้า, รหัส PIN, รหัสผ่าน หรือรูปแบบ (pattern) ของอุปกรณ์

## 2. จุดแข็งในปัจจุบัน

### ความปลอดภัยและการจัดเก็บ

- ข้อมูลกระเป๋าเงินได้รับการเข้ารหัสในเครื่อง ซึ่งสอดคล้องกับเป้าหมายของตู้นิรภัยออฟไลน์
- ฟิลด์ที่ละเอียดอ่อน เช่น ไพรเวตคีย์และวลีเมล็ดพันธุ์ ได้รับการเข้ารหัสเพิ่มเติมในระดับฟิลด์ก่อนที่รายชื่อกระเป๋าเงินทั้งหมดจะถูกเข้ารหัส
- Android เนทีฟมีปลั๊กอิน Device Credential โดยเฉพาะ ซึ่งใช้ Android Keystore เพื่อห่อหุ้ม (wrap) กุญแจตู้นิรภัย
- ตั้งค่า `android:allowBackup="false"` ใน AndroidManifest เพื่อลดความเสี่ยงในการสำรองข้อมูลแอปโดยไม่ได้ตั้งใจ
- มีกลไก เช่น การล็อกอัตโนมัติเมื่อไม่มีการใช้งาน, การล้างคลิปบอร์ดอัตโนมัติ, โล่ความเป็นส่วนตัวเมื่อแอปพลิเคชันไม่ทำงาน และการขอกรอกรหัสผ่านหลักเมื่อดูข้อมูลที่ละเอียดอ่อน
- รองรับฟังก์ชัน wipe/reset เมื่อตู้นิรภัยพบข้อผิดพลาดร้ายแรง

### ประสบการณ์ผู้ใช้

- หน้าแรกรองรับรูปแบบหน้าจอแบบตอบสนอง (responsive) โดยแสดงรายการกระเป๋าเงินแบบหลายคอลัมน์บนหน้าจอขนาดใหญ่ และปรับให้เหมาะสมสำหรับโทรศัพท์มือถือ
- สามารถปรับขนาดการแสดงผลได้ตั้งแต่ 5% ถึง 200% เหมาะสำหรับอุปกรณ์ขนาดเล็กหรือผู้ใช้ที่ต้องการดูข้อมูลมากขึ้น
- มีโหมดแบบหนาแน่น/กะทัดรัด/กะทัดรัดพิเศษ (dense/compact/ultra compact) สำหรับรายการกระเป๋าเงิน
- ปุ่มสำหรับการคัดลอก, QR, ขยายกระเป๋าเงิน, เพิ่มกระเป๋าเงิน, เครื่องมือ, ค้นหา, ตัวกรอง และการจัดเรียงถูกวางไว้ใกล้กับขั้นตอนการทำงานจริง
- มีโฟลเดอร์สำหรับกระเป๋าเงิน vanity, ป้ายกำกับ NEW, วงแหวนเรืองแสงสำหรับกระเป๋าเงินที่สร้างขึ้นใหม่ และการนำทางอัตโนมัติไปยังโฟลเดอร์ที่บรรจุกระเป๋าเงินใหม่
- หน้าต่างแก้ไขยอดคงเหลือรวมถึงการค้นหา, วาง, คัดลอกที่อยู่, นำเข้า CSV, ตัวกรอง และบันทึกแบบร่างอัตโนมัติ
- Toasts/หน้าต่างยืนยัน ได้รับการปรับปรุงให้ดูเป็นมืออาชีพมากขึ้น และมีแนวโน้มที่จะปรับขนาดตามอัตราส่วนหน้าจอ

### คุณสมบัติ

- สร้างกระเป๋าเงินปกติ, นำเข้าด้วยตนเอง, และสร้างกระเป๋าเงิน vanity โดยใช้เวิร์กเกอร์ (worker) แยกต่างหาก
- สำรองข้อมูล `.xkey`, นำเข้า/ส่งออก CSV, ตัวตรวจจับข้อมูลซ้ำ, การวิเคราะห์ข้อมูล, เครื่องมือขั้นสูง
- การโอนผ่าน QR ที่ป้องกันด้วยรหัสผ่าน, เครื่องสแกน QR, แชร์/ดาวน์โหลด QR
- รองรับเครือข่ายยอดนิยม: XLAYER, ETH, BSC, Polygon, Arbitrum, Optimism, Solana, Tron, Base
- รองรับ 15 ภาษา
- ดึงเวอร์ชันจาก `package.json`/ข้อมูลแอปเนทีฟ และแสดงในแอป

### บิลด์และ Android

- `npm run lint` ทำงานเสร็จสมบูรณ์
- `npm run build` ทำงานเสร็จสมบูรณ์
- `npx cap sync android` ซิงค์ไฟล์เว็บไปยัง Android สำเร็จ
- ปัจจุบันเวอร์ชัน Android คือ `versionName "5.7.0"` และ `versionCode 57`
- `.gitignore` ละเว้นไฟล์ที่ไม่จำเป็นอย่างเหมาะสม เช่น `1/`, สิ่งประดิษฐ์ (artifacts) ของการสร้างแอป, รหัสความลับของการลงนาม (signing secrets), `.xkey`, APK/AAB และไฟล์ภายในเครื่อง

## 3. จุดอ่อนและปัญหาที่อาจเกิดขึ้น

### ระดับสูง

1. การอ้างอิงอ็อบเจ็กต์ (Dependencies) มีคำเตือนด้านความปลอดภัยจาก `npm audit`

   การรัน `npm audit --omit=dev` รายงานว่า:

   - `vite 8.0.0 - 8.0.15`: มีความรุนแรงสูง, เกี่ยวข้องกับพาธของ Windows/UNC ในเซิร์ฟเวอร์การพัฒนา
   - `ws` ผ่าน `ethers`: มีความรุนแรงสูง/ปานกลาง `npm audit fix --force` เสนอให้ลดเวอร์ชันของ `ethers` ไปเป็นเวอร์ชันหลัก 5 ซึ่งอาจทำให้เกิดการเปลี่ยนแปลงที่ส่งผลกระทบให้เกิดข้อบกพร่อง (breaking changes)

   ข้อเสนอแนะ: ให้อัปเดต Vite อย่างปลอดภัยภายในช่วง patch/minor ก่อน สำหรับ `ethers/ws` ให้ตรวจสอบเวอร์ชันใหม่ของ `ethers` หรือเขียนทับ `ws` หาก upstream รองรับ หลีกเลี่ยงการใช้ `--force` อย่างสุ่มสี่สุ่มห้า

2. รีลีสของ Android ไม่ได้เปิดใช้งาน shrink/minify

   ในปัจจุบัน `android/app/build.gradle` ระบุ `release { minifyEnabled false }` สิ่งนี้ไม่ทำให้แอปแครช แต่จะทำให้ไฟล์ APK/AAB ถูกทำวิศวกรรมย้อนกลับ (reverse engineering) ได้ง่ายขึ้นและมีขนาดใหญ่ขึ้น

   ข้อเสนอแนะ: พยายามเปิดใช้งาน R8/ProGuard สำหรับการรีลีส, เพิ่ม keep rules สำหรับ Capacitor/ปลั๊กอินหากจำเป็น และทดสอบอย่างละเอียดก่อนเผยแพร่

3. คีย์ AES ສຳรอง (Fallback) ยังคงถูกเก็บไว้ใน Preferences

   ปัจจุบันโค้ดเก็บ `xkey_aes_fallback` ไว้สำหรับการกู้คืนหรือความเข้ากันได้กับเว็บ/ระบบสำรอง นี่เป็นการประนีประนอมเพื่อลดความเสี่ยงที่ตู้นิรภัยจะสูญหายเมื่อเปลี่ยนวิธีการล็อกหน้าจอ แต่ในด้านความปลอดภัยขั้นพื้นฐานของ Android แล้ว จะอ่อนแอกว่าการเก็บคีย์ไว้ใน Keystore เพียงอย่างเดียว

   ข้อเสนอแนะ: แบ่งแยกทั้งสองโหมดอย่างชัดเจน:
   - โหมดปลอดภัยของ Android (Android Secure Mode): คีย์จะถูกปลดล็อก (unwrapped) ผ่าน Keystore/Device Credential เท่านั้น
   - โหมดความเข้ากันได้ (Compatibility Mode): เก็บรักษาคีย์สำรองไว้ พร้อมแสดงคำเตือนให้ผู้ใช้ทราบอย่างชัดเจน

4. งานแปลรองบางรายการยังคงเป็นภาษาอังกฤษ

   การตรวจสอบอัตโนมัติแสดงให้เห็นว่าหลายภาษา (locale) เช่น `de`, `fr`, `es`, `hi`, `id`, `pt`, `tr`, `ar`, `th` ยังคงมีสตริงเช่น `Remove master password?`, `Enter master password`, `Wrong password`, `Pinned`, `Unpin`, `Double AES-256 with biometrics`

   ข้อเสนอแนะ: สร้างสคริปต์ตรวจสอบ i18n ใน CI เพื่อทำให้บิลด์ล้มเหลวหาก locale ขาดคีย์บางตัว หรือยังมีคีย์ดิบที่สำคัญ (raw keys) หลงเหลืออยู่

### ระดับกลาง

1. ขาดคีย์ในหลายภาษา (locale)

   เมื่อเทียบกับ `en.js` ภาษาส่วนใหญ่ยกเว้น `vi` ขาดคีย์ต่อไปนี้:
   - `common.warning`
   - `createWallet.vanityLongTitle`

   เนื่องจาก `LanguageContext` มีการตั้งสำรองข้อมูลไว้เป็นภาษาอังกฤษ แอปจึงไม่ขัดข้อง แต่ประสบการณ์การใช้งานหลายภาษายังไม่สมบูรณ์

2. `chainBulk` เป็นคีย์ส่วนเกินในหลาย locale

   หลาย locale มีกลุ่ม `chainBulk.*` แต่ไม่มีใน `en.js` นี่อาจเป็นคีย์เก่าหรือคีย์ที่ยังไม่ได้ซิงก์ แม้จะไม่ได้ทำให้เกิดข้อผิดพลาดโดยตรง แต่ก็ทำให้การจัดการงานแปลยากขึ้น

3. โหมด CryptoJS AES passphrase ไม่ใช่มาตรฐานการเข้ารหัสที่ทันสมัยที่สุด

   `CryptoJS.AES.encrypt(data, key)` สามารถทำงานได้ แต่ไม่อธิบายชัดเจนเท่าโมเดลมาตรฐานที่มีการกำหนดแท็ก salt/KDF/IV/auth แยกต่างหาก AES-GCM หรือ WebCrypto จะง่ายต่อการตรวจสอบมากกว่า

   ข้อเสนอแนะในระยะยาว: ย้ายรูปแบบของตู้นิรภัยไปเป็น WebCrypto AES-GCM พร้อมกำหนดพารามิเตอร์ PBKDF2/Argon2id อย่างชัดเจน, มีเพย์โหลดแบบระบุเวอร์ชัน และบังคับใช้แท็กการตรวจสอบสิทธิ์ (authentication tags)

4. รหัสผ่านหลักใช้ PBKDF2 โดยมีการวนซ้ำ (iterations) 10,000 รอบ

   ระดับนี้ถือว่าค่อนข้างต่ำเมื่อเทียบกับมาตรฐานในปัจจุบันสำหรับการปกป้องข้อมูลที่ละเอียดอ่อน แม้ว่านี่จะเป็นรหัสผ่านรองสำหรับดูไพรเวตคีย์/วลีเมล็ดพันธุ์ และไม่ใช่กุญแจหลักของตู้นิรภัย แต่ก็ควรเพิ่มจำนวนขึ้น

   ข้อเสนอแนะ: เพิ่มจำนวนการวนซ้ำตามเกณฑ์มาตรฐาน (benchmarks) ของอุปกรณ์ และเก็บแฮชของเวอร์ชัน (version hash) ไว้ เพื่อป้องกันการโยกย้ายข้อมูลไม่ให้ข้อมูลเก่าเสียหาย

5. การล้างคลิปบอร์ดอัตโนมัติไม่ได้รับประกันแบบ 100% ในทุกแพลตฟอร์ม

   โค้ดได้ตรวจสอบว่าคลิปบอร์ดมีค่าที่ถูกต้องหรือไม่ก่อนจะล้าง ซึ่งเป็นวิธีการที่ดี อย่างไรก็ตาม เบราว์เซอร์/Android อาจจำกัดการเขียนคลิปบอร์ดหากไม่ใช่การกระทำที่เกิดจากผู้ใช้โดยตรง

   ข้อเสนอแนะ: ควรอธิบายอย่างชัดเจนใน UI ว่า "xKey จะพยายามล้างคลิปบอร์ดหากระบบปฏิบัติการอนุญาต" โดยไม่ต้องสัญญาอย่างเด็ดขาด

6. ขาดการทดสอบอัตโนมัติในทางปฏิบัติ

   โครงการนี้มีการทำงานของ lint และ build แต่ขาดการทดสอบ unit/e2e สำหรับขั้นตอนสำคัญ เช่น การปลดล็อก, นำเข้า/ส่งออก, สร้างกระเป๋าเงิน vanity, แก้ไขยอดเงิน, จัดการคลิปบอร์ด และ i18n

   ข้อเสนอแนะ: เพิ่ม smoke tests โดยใช้ Playwright สำหรับเว็บไซต์ และสร้างเช็กลิสต์สำหรับขั้นตอนของ Android/การทดสอบการรีลีสด้วยตนเอง

### ระดับต่ำ

1. `console.error` ยังปรากฏอยู่ในบางพื้นที่

   ไม่สำคัญนัก แต่ควรรวมไว้ใน logger หรือสภาพแวดล้อมที่ใช้สำหรับนักพัฒนาเท่านั้น เพื่อหลีกเลี่ยงการรั่วไหลของ stacks ที่ไม่จำเป็นใน production

2. Vite แสดงคำเตือนเกี่ยวกับ chunks ที่มีขนาดใหญ่

   `index` และ `scan` chunks มีขนาดใหญ่ สิ่งนี้ไม่ใช่ปัญหาเมื่อรันแอปพลิเคชัน (runtime error) แต่ทำให้โหลดแอปพลิเคชันช้าในอุปกรณ์สเปคต่ำได้

   ข้อเสนอแนะ: ใช้ Lazy load สำหรับเครื่องสแกน QR, หน้าต่างที่ใช้ ethers หนักๆ, เครื่องมือขั้นสูง และการดูแดชบอร์ดที่ลึกขึ้น

3. UI บางส่วนอาจเกิดความผิดพลาดได้เมื่อสเกลจอเล็กมากหรือใหญ่มาก

   แอปพลิเคชันสามารถรองรับสเกลจอได้ดีในหลายๆ จุด แต่องค์ประกอบขนาดใหญ่ ได้แก่ กรอบข้อความโต้ตอบ (modal), คิวอาร์โค้ด, แบบฟอร์มแบบหนาแน่น, แผ่นควบคุมล่าง (bottom sheets) และบัตรข้อมูลกระเป๋าเงิน จำเป็นต้องได้รับการทดสอบที่สเกล 5%, 50%, 75%, 100%, 150% และ 200%

## 4. การประเมินกลุ่มฟีเจอร์

### ความปลอดภัยของการปลดล็อก

แนวทางปฏิบัติของ Android Device Credential ถือเป็นสิ่งที่ถูกต้อง เนื่องจาก OS จะจัดการชีวมิติและใช้รหัส PIN/รหัสผ่าน/รูปแบบหน้าจอเป็นฟอลแบ็ก ความเสี่ยงหลักอยู่ที่การย้ายข้อมูลระหว่างระบบ PIN รุ่นเก่า คีย์ที่เป็นค่าสำรอง (fallback keys) และ Keystore keys

ข้อเสนอแนะ:
- แสดงหน้าจอ "สถานะความปลอดภัยของตู้นิรภัย": Android Secure, Web Fallback, Compatibility และระบุถึงความจำเป็นของการตั้งค่าล็อกเครื่อง
- หากพบกุญแจที่ไม่ถูกต้อง ไม่ควรสร้างกุญแจตู้นิรภัยขึ้นมาใหม่ทันทีหากยังมีข้อมูลเข้ารหัสอยู่ในตู้นิรภัยอันเดิม แต่ควรแนะนำให้ผู้ใช้ทำการกู้คืน/ล้างข้อมูลแทน
- เก็บบันทึกสถานะการปลดล็อกภายในระบบโดยไม่ให้ข้อมูลสำคัญถูกบันทึก

### การสร้าง Vanity Walle

การใช้ worker ต่างหากถือว่าถูกต้อง เนื่องจากเป็นการป้องกันไม่ให้ UI ค้าง การอัปเกรดล่าสุด เช่น การกำหนดจำนวนกระเป๋าเงิน, บันทึกอัตโนมัติไปยังโฟลเดอร์, หยุดการล็อกอัตโนมัติไว้ชั่วคราวระหว่างสร้าง, กำหนดเวลา และคำเตือนเมื่อรูปแบบหน้าจอยาวเกินไป ถือว่าสมเหตุสมผลแล้ว

ข้อเสนอแนะ:
- แสดงความน่าจะเป็นและเวลาโดยประมาณให้ชัดเจนตามความยาวของแพตเทิร์น
- ควรอนุญาตให้หยุดชั่วคราว/ทำงานต่อ/ยกเลิกได้
- เก็บประวัติงานที่สร้างแล้วเพื่อให้ผู้ใช้ทราบว่ากระเป๋าเงินแต่ละใบมาจากคิวงานกลุ่มใด
- แสดงคำเตือนแบบเด่นชัดสำหรับแพตเทิร์นที่ยาวเกินไปในมือถือ

### การแก้ไขยอดคงเหลือของสินทรัพย์

กระบวนการนี้เหมาะสำหรับผู้ใช้ที่ต้องการตรวจยืนยันที่อยู่ผ่านบล็อกเอกซ์พลอเรอร์ แล้วค่อยกรอกข้อมูลยอดคงเหลือเองด้วยตนเอง ฟีเจอร์ที่โดดเด่นมีทั้งการค้นหา, คัดลอกที่อยู่, วางที่อยู่, ฟิลเตอร์กรองข้อมูล, รองรับการดึงข้อมูลจากไฟล์ CSV และฟีเจอร์บันทึกร่างข้อมูลอัตโนมัติ

ข้อเสนอแนะ:
- เพิ่มโหมด "การตรวจสอบทีละขั้นตอน": หน้าจอแสดงผลทีละ 1 กระเป๋าเงิน มีข้อมูลที่อยู่แบบครบถ้วน, ปุ่มคัดลอก, ลิงก์เข้าสู่เอกซ์พลอเรอร์แยกตามเครือข่าย พร้อมช่องสำหรับกรอกข้อมูลขนาดใหญ่
- ควรเพิ่มฟังก์ชัน "ตรวจสอบแล้ว" เพื่อป้องกันการข้ามการป้อนข้อมูล
- ยอมรับการนำเข้าไฟล์ CSV ผ่านทางคอลัมน์ `address,balance,unit,network`
- เพิ่มฟังก์ชันยกเลิก (undo) หลังการแก้ไขข้อมูลครั้งล่าสุด

### รองรับการทำงานหลายภาษา

การสลับกลับ (Fallback) ไปใช้ภาษาอังกฤษจะช่วยป้องกันไม่ให้ระบบ UI เสียหาย แต่แอปพลิเคชันที่มีกลุ่มเป้าหมายระดับโลกก็ยังต้องการให้มีการบริหารจัดการระบบภาษาที่รัดกุมมากขึ้น

ข้อเสนอแนะ:
- สร้างสคริปต์ทำงานร่วมกับคำสั่ง `npm run i18n:check`
- แจ้งเตือนเมื่อมีคีย์หายไป, คีย์ข้อมูลที่ไม่จำเป็น หรือปุ่มแปลภาษาแบบ raw keys ลงใน UI
- จัดลำดับความสำคัญของระบบแปลที่ต้องมีความแม่นยำ ได้แก่ ข้อมูลกลุ่มความปลอดภัยของตู้นิรภัย, สร้างแบ็กอัป, ยืนยันการลบไพรเวตคีย์, ไพรเวตคีย์ และ seed phrase

### ระบบปฏิบัติการ Android รุ่นใหม่

องค์ประกอบในรุ่นล่าสุดเพียงพอสำหรับการสร้าง (build) ระบบและการซิงโครไนซ์แล้ว แต่ต้องเพิ่มเติมเรื่องการเสริมเกราะเพื่อลดช่องโหว่ความเสี่ยง (release hardening) เข้ามา

ข้อเสนอแนะ:
- เปิดระบบการหดย่อขนาดไฟล์ (minify) ให้กับการออกแอปหลังเสร็จสิ้นกระบวนการตรวจสอบ
- เพิ่มคำสั่งเข้า CI คือ `npm audit --omit=dev` และระบุระบบบัญชีผู้รับรองเข้าด้วยกัน
- เพิ่มช่องทางการใช้งาน (GitHub Actions) เมื่อต้องพุชการแก้ไขของแอปไปยังสโตร์ (tag pushes) ที่รองรับรหัสฐานข้อมูลแบบ APK/AAB
- ใส่เลขบอกข้อมูลบันทึกรีลีสในคลังเก็บข้อมูล

## 5. เสนอแนวความคิดของการอัปเกรด

### ในระยะสั้น

- อัปเดตข้อมูลการแก้ไขการแปลภาษาที่หายไป ได้แก่: `common.warning`, `createWallet.vanityLongTitle`
- สะสางข้อมูลที่พิมพ์ด้วยภาษาอังกฤษที่ปะปนในโค้ดระบบภาษาอื่นๆ
- ใส่สคริปต์ตรวจสอบระบบคำสั่ง i18n เข้าไว้ที่ CI
- อัปเดตส่วนระบบคำสั่ง Vite โดยตอบสนองต่อคำแนะนำทางระบบในปัจจุบัน
- ใส่ปุ่มหน้าระบบประเมินความปลอดภัย "สถานะความปลอดภัย" (Security Status) ที่ตั้งค่าระบบ
- ระบุหมายเหตุสำหรับการแจ้งให้คลิปบอร์ดทำงานได้เต็มที่
- กำหนดปุ่มแสดงคำสั่ง "เปิดในเอกซ์พลอเรอร์" ตามโครงข่ายโหมดในป๊อปอัพเมนูโชว์ข้อมูลเครดิต
- นำเอาระบบแจ้งเตือน "แถบรายการระบบย้อนกลับ (Undo snackbars)" เข้าบรรจุด้วย เพื่อโชว์รายการว่ากระเป๋าถูกลบไปแล้ว, บัญชีแสดงยอดดุลคงค้างถูกนำกลับมาแก้ไข ตลอดจนโฟลเดอร์ถูกเปลี่ยนเป็นอย่างอื่น

### ระยะปานกลาง

- ทำระบบโยกย้ายไฟล์ที่มีการอิงตามการทำแพ็กเกจอัปเดตรูปแบบ WebCrypto AES-GCM
- ปรับแยกการทำงานในส่วนของระบบ Android Security ให้แยกขาดจาก Compatibility Mode อย่างเด็ดขาด
- ใส่ชุดโค้ด smoke tests ตรวจสอบความถูกต้องที่ส่วนฟอร์มโครงหลักของ Playwrigh
- ลดขนาดชุดบรรจุข้อมูลช่วงเริ่มต้นโดยกำหนดรูปแบบโหมดระบบเป็น "โหลดเฉพาะข้อมูลที่ต้องใช้ตอนเปิด" (Lazy load)
- เพิ่มระบบให้ผู้ใช้งานอิมพอร์ตระบบเข้าไปในโหมดที่ไม่กระทบกับข้อมูลการเก็บล็อก
- ยกระดับรูปแบบให้สามารถสั่งทำงานฟีเจอร์ โหมดแอป "Vault Audit" ซึ่งใช้ตรวจสอบดูช่องโหว่ อาทิ การที่กระเป๋าถูกทิ้งร้าง การลงข้อมูลแอดเดรสหลายที่ การบันทึกเชื่อมระบบไร้ศูนย์กลางผิดพลาด ลืมการป้อนชื่อ หรือมีชื่อเจ้าของกระเป๋าไม่ตรงกับการใช้

### ระยะยาว

- จัดตั้งหน่วยแนะนำวิธีการแก้ไขสำหรับสถานการณ์ต่างๆ อาทิ แก้เปลี่ยนรหัสเครื่อง โดนบังคับให้ต้องเปลี่ยนช่องทางการตรวจลายนิ้วมือ รวมถึงวิธีการเก็บระบบข้อมูลสำรอง `.xkey`
- ออกแบบแอปให้สามารถถ่ายทอดสิทธิได้เมื่อเปลี่ยนโทรศัพท์หลายเครื่องในแบบใช้รหัสเข้าร่วม
- กำหนดสเปกของการใช้งานแบบเข้ารหัสขั้นสูงรองรับการป้องกันด้วยกุญแจฮาร์ดแวร์เพื่อรับรองระบบปลอดภัยระดับมืออาชีพ
- มีกระบวนทัศน์สร้างเช็กซัมเครือข่ายสำหรับแอดเดรส
- ให้ความสำคัญกับการแจกเอกสารสำหรับการแบ็กอัปที่พิมพ์ออกมาได้ โดยครอบคลุมตั้งแต่แอดเดรส, หมายเลขเครือข่าย, ป้ายแสดงรายการบันทึกเตือน ซึ่งผู้ถือกระเป๋าเท่านั้นจะเป็นผู้ถือกุญแจเข้ารหัสนี้เอาไว้ได้
- รองรับการใช้งานผ่านส่วนฟังก์ชัน PWA หรือในส่วนแสดงผลบน PC กรณีต้องการให้แอปทำงานบนคอมแบบออฟไลน์

## 6. ทิศทางของแอปพลิเคชันนี้ในอนาคต

ตัวแอปพลิเคชันได้พุ่งความสนใจไปที่รูปแบบการเป็นระบบจัดการตู้นิรภัยที่รวบรวม "ระบบบริหารและเก็บรักษาแบบออฟไลน์สำหรับผู้ที่มีกระเป๋าหลายบัญชี" ไม่ต้องการให้มันเข้าไปแข่งขันเป็นหน้าตาที่เชื่อมธุรกรรมโดยตรง เพราะจะก่อความเสี่ยงในหลายๆ รูปแบบ ตั้งแต่เรื่องของระบบความปลอดภัยที่ถูกนำไปผูกโยงร่วมกับ RPC, ปัญหาไวรัสแฝงทางเมล์ (Phishing), ขอบข่ายของข้อกำหนดการทำกิจกรรมด้านธุรกิจที่ต้องพึ่งการลงรายเซ็น และก่อช่องโหว่ด้านรูปแบบทางเทคโนโลยีที่เป็นอันตราย

ทิศทางที่ควรเป็น:
1. ให้ความสำคัญของข้อมูลความปลอดภัย (Data safety): ดูแลระบบการเก็บรักษาฐานระบบ, เปิดประเมินการกู้ข้อมูล, วางระบบแจ้งเตือนต่างๆ ให้มีความถูกต้องแม่นยำ พร้อมเข้าตรวจสอบข้อมูลของตู้นิรภัยอย่างเข้มงวด
2. จัดวางตัวเลือกเพื่อเพิ่มการรองรับระบบกระเป๋าที่มีหลายแบบให้สะดวกไวขึ้น: เข้าถึงช่องทางแยกโฟลเดอร์ได้เร็ว, ใส่คีย์ป้ายคำให้, กรองหมวดต่างๆ ได้เร็วไว, แก้ไขเป็นชุดข้อมูล (Batch edit) และพร้อมตอบสนองต่อการสแกน QR แบบทันที, ควบคุมรหัสอัปเดตสร้างบัญชีสวยๆ (vanity generation) ให้ตอบสนองรวดเร็ว
3. ทำให้การตอบสนองบนอุปกรณ์รุ่น Android นำพาระบบการยืนยันผู้เปิดด้วยฟีเจอร์ Device Credential, เข้ารหัสด้วย Keystore, ใช้หน่วยความจำในหน้าต่างแอป, ช่วยปรับแต่งดึงโอนถ่ายไฟล์แชร์ด้วย QR ได้ตลอด
4. ระบบตอบรับบนหน้าจอแอปมีพื้นที่โชว์เต็มประสิทธิภาพ: ขนาดอักษรกระชับ, ตัวอักษรอ่านง่าย, ตัวระบบรองรับโหมดของคนใช้แท็บเล็ตได้ดี, ขึ้นรูปตอบแสดงผลได้อย่างไม่กวนสายตา (short toasts) ตลอดจนโชว์กรอบบล็อกในส่วนอธิบายฟอร์มที่ไม่ขัดลูกตา
5. โชว์โครงหน้าโปรแกรมในรูปแบบโปร่งใสในข้อมูล: สามารถตรวจสอบข้อมูลล็อกความปลอดภัยของอุปกรณ์, รีวิวข้อกังวลในการใช้งาน รวมถึงแสดงโชว์ข้อมูลเวอร์ชันในแอป และวิธีการดูแลแบ็กอัปตลอดจนรีเซ็ตได้อย่างเปิดเผย

## 7. สรุปท้ายรายงานฉบับนี้

โปรเจกต์งานถือว่าเป็นเรื่องที่มีองค์ประกอบต่างๆ ค่อนข้างมั่นคง ไม่ว่าจะเป็นระบบฟีเจอร์, แผนนำทางในกระบวนการทำงานรูปแบบเข้าแอปออฟไลน์, สามารถส่งให้เครื่องในแพลตฟอร์มของ Android เชื่อมรับหน้าจอได้, แอปผ่านการขัดเกลามาในหน้าเว็บที่มีความละเอียดของสมาร์ตโฟน, รวมทั้งสามารถบริหารหน้าต่างที่มีระบบของฟังก์ชันตัวควบคุมได้อย่างรัดกุม

ดังนั้นแนวทางในการผลักดันเป้าหมายในลำดับความสำคัญอันดับต่อไปจึงไม่ได้เน้นว่าผู้ปรับปรุงฟังก์ชันต้องรีบใส่ฟีเจอร์ฟังก์ชันในส่วนฟอร์มที่มีให้ทันท่วงที แต่แอปต้องวางแนวทางการเพิ่มส่วนฟังก์ชันที่ออกแบบมาเพื่อสกัดส่วนอันตรายต่างๆ ที่จะเป็นข้อผิดพลาดหรือเกิดช่องโหว่ต่างๆ ซึ่งประกอบไปด้วย:
- รีบใส่ระบบแอดเดรสสำหรับรองรับหลายๆ ภาษาให้ผ่านฉลุยไปตามเป้าหมายของ i18n
- เน้นให้ผู้ทดสอบทำการอัปเดตแอปแอนดรอยด์ลงรุ่นให้ถูกต้อง
- ยกระดับแนวทางการออกแบบและระบุรหัสให้ Keystore/ระบบตัวเลือกข้อมูลสำรอง (fallback) ของแอปต้องแยกเด็ดขาด
- เสริมระบบในการเขียนแบบทดสอบสำหรับการเข้าจัดการรูปแบบระบบข้อมูลแบบอัตโนมัติ
- บังคับดูแลการเชื่อมโยงโครงสร้างที่มีผลกระทบกับความเปลี่ยนแปลงส่วน Audit Dependencies
- ให้ผู้กำหนดค่าวางโครงรหัสแบบเปิดฟังก์ชันพื้นฐานเพื่อให้รูปแบบอิงระบบโค้ดได้แบบนานที่สุด

หากมองจากแนวทางและเป้าประสงค์เหล่านี้ แล้วสามารถตอบสนองให้แอปออกมาดูดี มันจะกลายไปเป็นหนึ่งในตู้เซฟแห่งความหวังที่มีความมั่นใจสำหรับให้สายเหรียญต่างๆ ได้หยิบไปคอยสับเปลี่ยนดูแลกระเป๋าทั้งผองบน Web3 ได้อย่างราบรื่น
