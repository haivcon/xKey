# Отчет о проверке проекта xKey

Дата проверки: 2026-06-16
Текущая версия: 5.7.0
Область применения: исходный код React/Vite, Capacitor Android, хранение, безопасность, пользовательский интерфейс, многоязычность, сборка и направление развития продукта.

## 1. Цель проекта

xKey — это приложение для управления хранилищем (vault) кошельков Web3 с упором на офлайн-работу (offline-first). Приложение позволяет пользователям хранить адреса кошельков, приватные ключи, seed-фразы, заметки, метки, папки, QR-коды, файлы резервных копий `.xkey`, данные CSV и балансы активов локально на устройстве.

Основная цель xKey — стать локальным «хранилищем приватных ключей», а не онлайн-кошельком для транзакций. Пользователи могут использовать xKey для:

- Управления несколькими кошельками Web3 в зашифрованном хранилище.
- Хранения приватных ключей и seed-фраз в локальном зашифрованном формате.
- Создания новых кошельков, ручного импорта кошельков, генерации красивых (vanity) кошельков по префиксу/суффиксу.
- Группировки кошельков по папкам, меткам, сетям, статусу закрепления или балансам.
- Резервного копирования/восстановления с использованием защищенных паролем файлов `.xkey`.
- Экспорта CSV при необходимости проведения инвентаризации или аудита.
- Сканирования, отображения, отправки и загрузки QR-кодов для адресов или данных кошелька.
- Ручного отслеживания балансов в дополнительных единицах, таких как `$`, `USDT`, `VND`, `CNY`, `KRW`, `JPY`, `EUR`, `RUB`, `INR`, баллах или пользовательских метках.
- Использования Android Device Credential для разблокировки хранилища с помощью отпечатка пальца, лица, PIN-кода, пароля или графического ключа устройства.

## 2. Текущие преимущества

### Безопасность и хранение данных

- Данные кошелька зашифрованы локально, что соответствует цели создания офлайн-хранилища.
- Конфиденциальные поля, такие как приватные ключи и seed-фразы, дополнительно шифруются на уровне поля перед шифрованием всего списка кошельков.
- В нативном Android есть специальный плагин Device Credential, который использует Android Keystore для упаковки (wrap) ключа хранилища.
- В AndroidManifest установлено `android:allowBackup="false"`, что снижает риск непреднамеренного резервного копирования данных приложения.
- Включает такие механизмы, как автоблокировка при неактивности, автоочистка буфера обмена, экран конфиденциальности, когда приложение неактивно, и запрос мастер-пароля при просмотре конфиденциальных данных.
- Поддерживает удаление/сброс данных (wipe/reset), когда в хранилище возникает критическая ошибка.

### Пользовательский опыт

- Главная страница поддерживает адаптивный дизайн с многоколоночным списком кошельков на больших экранах и оптимизацией для мобильных устройств.
- Имеет настраиваемое масштабирование отображения от 5% до 200%, что подходит для небольших устройств или пользователей, желающих просматривать больше данных.
- Предоставляет плотный (dense)/компактный (compact)/сверхкомпактный (ultra compact) режимы для списка кошельков.
- Кнопки для копирования, QR, развертывания кошелька, добавления кошелька, инструментов, поиска, фильтрации и сортировки расположены близко к фактическому рабочему процессу.
- Включает папку для красивых (vanity) кошельков, метки NEW, светящееся кольцо для недавно созданных кошельков и автоматическую навигацию в папку, содержащую новый кошелек.
- Модальное окно редактирования баланса включает поиск, вставку, копирование адреса, импорт CSV, фильтр и автосохранение черновиков.
- Всплывающие уведомления (Toasts)/подтверждения были переработаны, чтобы выглядеть более профессионально, и масштабируются в соответствии с соотношением сторон дисплея.

### Особенности

- Создание обычных кошельков, ручной импорт и генерация красивых кошельков (vanity) с использованием выделенного воркера (worker).
- Резервное копирование `.xkey`, импорт/экспорт CSV, детектор дубликатов, аналитика, расширенные инструменты.
- Передача QR с защитой паролем, сканер QR, отправка/скачивание QR.
- Поддерживает популярные сети: XLAYER, ETH, BSC, Polygon, Arbitrum, Optimism, Solana, Tron, Base.
- Поддержка многоязычности с 15 языками.
- Версия извлекается из `package.json`/информации о нативном приложении и отображается в приложении.

### Сборка и Android

- `npm run lint` завершается успешно.
- `npm run build` завершается успешно.
- `npx cap sync android` успешно синхронизирует веб-ресурсы с Android.
- Текущая версия Android — `versionName "5.7.0"` и `versionCode 57`.
- `.gitignore` правильно исключает `1/`, артефакты сборки, секреты подписи, `.xkey`, APK/AAB и локальные файлы.

## 3. Недостатки и потенциальные проблемы

### Высокий уровень

1. Зависимости имеют предупреждения о безопасности от `npm audit`.

   Запуск `npm audit --omit=dev` сообщает:

   - `vite 8.0.0 - 8.0.15`: высокая степень серьезности, связано с путями Windows/UNC в dev-сервере.
   - `ws` через `ethers`: высокая/умеренная серьезность. `npm audit fix --force` предлагает откатить `ethers` до мажорной версии 5, что может вызвать критические изменения (breaking changes).

   Рекомендация: сначала безопасно обновите Vite в пределах патча/минорной версии. Для `ethers/ws` проверьте наличие более новой версии `ethers` или переопределите `ws`, если это поддерживается upstream; избегайте слепого использования `--force`.

2. В релизе для Android не включены shrink/minify.

   В `android/app/build.gradle` в настоящее время установлено `release { minifyEnabled false }`. Это не приводит к сбою приложения, но облегчает обратную разработку (reverse engineering) APK/AAB и увеличивает его размер.

   Рекомендация: Попробуйте включить R8/ProGuard для релиза, при необходимости добавьте правила сохранения (keep rules) для Capacitor/плагинов и тщательно протестируйте перед публикацией.

3. Резервный (Fallback) ключ AES по-прежнему хранится в Preferences.

   Код в настоящее время сохраняет `xkey_aes_fallback` для восстановления или совместимости с вебом/резервным вариантом. Это компромисс, чтобы снизить риск потери хранилища при изменении методов блокировки устройства, но с точки зрения нативной безопасности Android это слабее, чем хранение ключа только в Keystore.

   Рекомендация: Четко разделите два режима:
   - Безопасный режим Android: Ключ извлекается только через Keystore/учетные данные устройства.
   - Режим совместимости: Сохраняет резервный ключ с четким предупреждением для пользователя.

4. Некоторые второстепенные переводы все еще содержат английские строки.

   Автоматические проверки показывают, что во многих локалях, таких как `de`, `fr`, `es`, `hi`, `id`, `pt`, `tr`, `ar`, `th`, все еще есть строки типа `Remove master password?`, `Enter master password`, `Wrong password`, `Pinned`, `Unpin`, `Double AES-256 with biometrics`.

   Рекомендация: Создайте сценарий проверки i18n в CI для сбоя сборки, когда в локали отсутствуют ключи или все еще присутствуют важные необработанные (raw) ключи.

### Средний уровень

1. Отсутствуют ключи в локалях для нескольких языков.

   По сравнению с `en.js`, в большинстве локалей, кроме `vi`, отсутствуют:
   - `common.warning`
   - `createWallet.vanityLongTitle`

   Поскольку `LanguageContext` возвращается к английскому языку, приложение не дает сбой, но многоязычный интерфейс неполный.

2. `chainBulk` — это лишний ключ во многих локалях.

   Во многих локалях есть группа `chainBulk.*`, а в `en.js` ее нет. Это могут быть устаревшие или несинхронизированные ключи. Хотя они не вызывают прямых ошибок, они усложняют управление переводами.

3. Режим пароля CryptoJS AES не является самым современным стандартом шифрования.

   `CryptoJS.AES.encrypt(data, key)` работает, но он не так очевиден, как стандартная модель с выделенными тегами salt/KDF/IV/auth. AES-GCM или WebCrypto было бы проще проверять (аудит).

   Долгосрочная рекомендация: Перенесите формат хранилища на WebCrypto AES-GCM с явно определенными параметрами PBKDF2/Argon2id, версионированными полезными нагрузками (payloads) и обязательными тегами аутентификации.

4. Мастер-пароль использует PBKDF2 с 10 000 итераций.

   Этот уровень несколько низок по сегодняшним стандартам защиты конфиденциальных данных. Хотя это вторичный пароль для просмотра приватных ключей/seed-фраз, а не основной ключ хранилища, его все равно следует увеличить.

   Рекомендация: Увеличьте количество итераций на основе бенчмарков устройств и сохраняйте хеш версии, чтобы миграции не повреждали старые данные.

5. Автоматическая очистка буфера обмена не гарантируется на 100% на всех платформах.

   Код проверяет, содержит ли буфер обмена по-прежнему правильное значение перед его очисткой, что является хорошим подходом. Однако Android/браузеры могут ограничивать запись в буфер обмена, если она не вызвана жестом пользователя.

   Рекомендация: Четко опишите в пользовательском интерфейсе, что «xKey попытается очистить буфер обмена, если операционная система позволит», без абсолютных обещаний.

6. Отсутствие практических автоматизированных тестов.

   В проекте есть линтинг и сборка, но отсутствуют юнит/e2e тесты для критических процессов, таких как разблокировка, импорт/экспорт, создание vanity кошелька, редактирование баланса, операции с буфером обмена и i18n.

   Рекомендация: Добавьте дымовые тесты (smoke tests) с использованием Playwright для веб-версии и контрольный список (checklist) для инструментария Android/ручного тестирования релиза.

### Низкий уровень

1. `console.error` остался в нескольких местах.

   Не критично, но следует сгруппировать в логгер или среду только для разработки (dev-only), чтобы избежать утечки ненужных стеков (stacks) в продакшене.

2. Vite предупреждает о больших кусках кода (chunks).

   Куски кода `index` и `scan` большие. Это не ошибка во время выполнения, но это может замедлить загрузку приложения на слабых устройствах.

   Рекомендация: Ленивая загрузка (Lazy load) для сканера QR, тяжелых путей ethers, расширенных инструментов и более глубоких представлений панели управления.

3. Некоторые элементы пользовательского интерфейса могут сломаться при очень низких или очень высоких коэффициентах масштабирования.

   Приложение хорошо масштабируется во многих областях, но большие модальные окна, QR-коды, плотные формы, нижние панели (bottom sheets) и карточки кошельков все еще нуждаются в тестировании при 5%, 50%, 75%, 100%, 150% и 200%.

## 4. Оценка группы функций

### Безопасность разблокировки

Подход с использованием Android Device Credential верен, так как он позволяет ОС управлять биометрией и переходить на PIN-код/пароль/графический ключ. Основной риск заключается в миграции между старым механизмом PIN-кода, резервными ключами и ключами Keystore.

Рекомендации:
- Предоставьте экран «Статус безопасности хранилища»: Android Secure, Web Fallback, Compatibility, требует настройки блокировки устройства.
- Если обнаружен недействительный ключ, не создавайте автоматически новый ключ хранилища, если старое хранилище все еще содержит зашифрованный текст; вместо этого направьте пользователя к восстановлению/удалению (recovery/wipe).
- Регистрируйте внутренние статусы разблокировки без логирования конфиденциальных данных.

### Генерация Vanity кошельков

Использование отдельного воркера — правильный шаг, поскольку это предотвращает зависание пользовательского интерфейса. Недавние обновления, такие как количество кошельков, автосохранение в папку, приостановка автоблокировки во время генерации, ограничения по времени и предупреждения о длинных паттернах, вполне разумны.

Рекомендации:
- Четко отображайте вероятность/расчетное время на основе длины паттерна.
- Разрешите приостанавливать/возобновлять/останавливать задачу.
- Сохраняйте историю сгенерированных задач, чтобы пользователи знали, какие кошельки относятся к какой партии.
- Обеспечьте строгие предупреждения для чрезмерно длинных паттернов на мобильных устройствах.

### Редактирование баланса активов

Текущий рабочий процесс подходит для пользователей, которые проверяют адреса в блокчейн-эксплорерах, а затем вводят балансы вручную. Сильные стороны включают поиск, копирование адреса, вставку, фильтр, поддержку CSV и автосохранение черновиков.

Рекомендации:
- Добавьте режим «пошаговой проверки»: на экране отображается 1 кошелек за раз, полный адрес, кнопка копирования, ссылка на эксплорер по сети и большое поле ввода.
- Разрешите помечать как «проверенный» (checked), чтобы не пропустить записи.
- Разрешите импорт CSV со столбцами `address,balance,unit,network`.
- Добавьте функцию отмены (undo) для последнего редактирования.

### Многоязычность

Откат на английский язык предотвращает поломку пользовательского интерфейса, но продукт, ориентированный на международную аудиторию, требует более строгого контроля перевода.

Рекомендации:
- Создайте скрипт `npm run i18n:check`.
- Сообщайте об отсутствующих ключах, лишних ключах и необработанных ключах перевода в пользовательском интерфейсе.
- Отдайте приоритет точным переводам для групп безопасности, резервного копирования, удаления, приватного ключа и seed-фразы.

### Релиз Android

Текущая конфигурация достаточна для сборки и синхронизации, но не хватает усиления безопасности (hardening) релиза.

Рекомендации:
- Включите minify для релиза после тестирования.
- Добавьте шаг CI `npm audit --omit=dev` с четким списком разрешений (allowlist).
- Собирайте APK/AAB через GitHub Actions при пушах тегов (tag pushes).
- Храните версионированные примечания к релизу (release notes) в репозитории.

## 5. Предлагаемые идеи обновления

### Краткосрочные

- Исправьте все отсутствующие ключи перевода: `common.warning`, `createWallet.vanityLongTitle`.
- Очистите оставшиеся английские строки в других локалях.
- Добавьте сценарий проверки i18n в CI.
- Обновите Vite, чтобы устранить текущее предупреждение об уязвимости.
- Добавьте страницу «Статус безопасности» в настройках.
- Добавьте четкое примечание о том, что автоматическая очистка буфера обмена выполняется по мере возможности (best-effort).
- Добавьте кнопку «открыть в эксплорере» по сети в модальном окне редактирования баланса.
- Добавьте снэкбары (snackbars) с отменой действия для удаления кошелька, редактирования балансов и изменений папок.

### Среднесрочные

- Перенесите формат шифрования на версионированный WebCrypto AES-GCM.
- Разделите безопасный режим Android (Secure Mode) и режим совместимости (Compatibility Mode).
- Добавьте дымовые тесты (smoke tests) Playwright для основных потоков (main flows).
- Добавьте ленивую загрузку сканера/аналитики/расширенных инструментов, чтобы уменьшить размер начального пакета (bundle).
- Добавьте экспорт/импорт настроек, исключающий конфиденциальные данные.
- Добавьте режим «Аудит хранилища»: кошельки без резервных копий, дублирующиеся адреса, отсутствующие сети, отсутствующие имена или приватные ключи, не совпадающие с адресами.

### Долгосрочные

- Создайте официальное руководство по восстановлению для сценариев, таких как смена устройств, смена блокировки экрана, потеря биометрии или файлов `.xkey`.
- Добавьте зашифрованную передачу данных между несколькими устройствами с помощью составных QR-кодов или временных файлов.
- Добавьте вариант хранения только на аппаратном уровне (hardware-backed) для пользователей, требующих высокой безопасности.
- Добавьте проверку адреса с помощью контрольной суммы (checksum)/сети.
- Предоставьте бумажные шаблоны для резервного копирования: адрес, сеть, заметки, без приватных ключей, если пользователь решит.
- Улучшите поддержку Desktop/PWA, если xKey будет использоваться в качестве десктопного офлайн-хранилища.

## 6. Будущее направление продукта

xKey должен следовать по пути «профессионального офлайн-хранилища для пользователей с большим количеством кошельков». Не стоит преждевременно превращать его в онлайн-кошелек для транзакций, поскольку это повышает риски безопасности, зависимости RPC, векторы фишинга, ответственность за подписание транзакций и поверхности атак.

Правильное направление:
1. Приоритет безопасности данных: резервное копирование, восстановление, миграция, четкие предупреждения, аудит хранилища.
2. Приоритет быстрого управления множеством кошельков: папки, теги, фильтры, пакетное редактирование, CSV, QR, генерация vanity кошельков.
3. Приоритет стабильных нативных функций Android: Device Credential, Keystore, управление буфером обмена, выбор файлов, совместное использование/скачивание QR.
4. Приоритет плотного, но понятного пользовательского интерфейса: масштабирование, компактный режим, адаптивные макеты для планшетов, короткие тосты (toasts), неблокирующие модальные окна.
5. Приоритет прозрачности: статус безопасности, примечания к выпуску, явное указание версии в приложении, руководства по резервному копированию и удалению данных.

## 7. Заключение

Проект имеет прочную основу: он обладает богатым функционалом, четким подходом offline-first, интеграция Android Credential идет по правильному пути, пользовательский интерфейс сильно оптимизирован для мобильных устройств/планшетов и имеет комплексный набор инструментов для управления кошельками.

Главный приоритет в продвижении вперед заключается не в добавлении множества новых функций, а в том, чтобы сделать приложение «более устойчивым к поломкам»:
- Завершить внедрение i18n.
- Усилить (Harden) релиз для Android.
- Прояснить модель безопасности Keystore/резервных ключей (fallback).
- Добавить автоматизированные тесты для критических потоков.
- Управлять аудитом зависимостей.
- Стандартизировать формат шифрования в долгосрочной перспективе.

Если эти проблемы будут успешно решены, xKey может стать высоконадежным инструментом офлайн-хранилища для пользователей, управляющих несколькими кошельками Web3.
